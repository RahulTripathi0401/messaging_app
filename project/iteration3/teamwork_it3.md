**Planning a successful team meeting**

Team meetings were considered successful if the goals for the meeting were reached. To make sure that a team meeting went to plan, the group in advance set out goals for the meeting e.g. need to fix pytests for the updated spec. As it was clear what problems needed to be tackled, we could  efficiently resolve them. Another component of a successful team meeting was updating each other on our individual progress, so that everyone was on the same page when it came to the amount of work to do. We also recorded the minutes for each meeting so we could later refer to it when planning future meetings to avoid reiterating the same points.

**Steps to resolve disputes / Challenges**

As a group it was imperative that we had a disagreement resolution process. An example of this was when we were deciding what stories were relevant and whether or not to disregard some stories. To overcome this, we wrote down the pros and cons of both sides. Each member looked over it and we collaboratively came up with the best way forward. Overall, disagreements were rare however it was important to have a method to resolve such conflicts.

When confronted with a technical challenge that an individual was unsure on how to solve, they would first ask the group for some guidance. If the problem could not be solved, we would read resources that would help us solve the problem. If the issue was more specifically related the spec i.e. store data, we would then ask a tutor for guidance.

Overall, throughout the entire implementation of iteration one, two and three, there was no major obstacle that we were not able to overcome. Through the management strategies stated above, we were able to complete the project.

**Sharing Code Base**

Throughout the project, we used the version control system git to help share and manage our code repositories. Each member created their own local branch in which they could make changes to the section of code assigned to them. As the changes were made on their local branch, it did not impact other people's changes hence eliminating conflicts. When a section was complete, we would all push our changes to the online repo where it could be later merged with the master branch. Another way in which we had multiple members work on the same code was through pair programming. Pair programming allowed members to share ideas and resolve issues more quickly when tackling a singular task. This proved beneficial to the group. On a larger scale, we would share our code during a team meeting so everyone was able to see what we had done.
